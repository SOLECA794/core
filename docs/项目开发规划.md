# 操作系统赛内核赛道 6月30日交付 全周期时间规划
基于当前4月8日的时间节点，距最终交付共**11周零3天（83天）**，结合你们「先定框架再填实现、学习开发并行、分模块里程碑管控、兜底防延期」的核心诉求，制定以下可落地、强容错的时间规划，完全匹配内核赛道5大核心模块的开发节奏。

## 一、核心规划原则（先对齐共识，避免后期失控）
1.  **框架先行，保底优先**：先敲定全系统模块接口与交互逻辑，再填实现细节；每个模块先做必选基础功能，再做加分高级功能，杜绝“为了优化拖垮整体进度”。
2.  **学-做闭环，拒绝纯脱产学习**：摒弃“学完再做”，采用「对应模块知识点学习→框架代码搭建→核心逻辑实现→单元测试闭环」的并行模式，边学边落地。
3.  **里程碑卡死，单模块不延期**：每个模块设置固定交付周期，周末必须完成该模块的测试与文档输出，单模块延期不得占用后续模块的开发时间，优先挤压缓冲期。
4.  **双缓冲兜底，防交付风险**：全流程设置2周专属缓冲期，提前规避bug修复、学习卡点、文档撰写等突发耗时，绝对不压缩最终提交期的时间。

## 二、分阶段全周期时间规划
### 第一阶段：方向定版&基础攻坚期（4月8日-4月21日，共2周）
**核心目标**：解决“不知道做什么、基底没定、基础不牢”的核心问题，完成选型、框架定版、开源demo复现，给后续开发锁死边界。

| :--- | :--- | :--- |

| 4月11日-4月14日（4天） | 开源最小内核demo复现工程、开发环境搭建完成 | 1. 完成选定开源demo的源码通读、编译运行、核心逻辑拆解，完成最小功能复现；2. 并行完成系统编程核心知识点学习，重点吃透中断、内存、进程的底层原理 |
| 4月15日-4月21日（7天） | 全系统框架设计文档、模块开发里程碑终版、每个模块的接口定义文档 | 1. 完成参赛作品的完整框架设计，明确模块间的交互逻辑、接口规范、数据结构，彻底锁死代码框架，后续只做填充实现；2. 完成5个模块的前置知识学习，每个模块至少吃透开源demo中的对应实现；3. 对齐最终的模块开发时间节点，全员达成共识 |

**核心目标**：打通全系统链路，跑通完整内核，完成必选功能闭环，迭代加分项，预留1周缓冲应对延期。
1.  第一周（6月3日-6月9日）：全系统联调，解决模块间的耦合bug，跑通从开机到用户态程序运行的完整流程，确保所有必选功能100%可用，形成最小可用完整内核。

## 三、风险防控（彻底解决“时间来不及、模块太多做不完”的核心顾虑）
1.  **底线交付机制**：每个阶段、每个模块都设置不可突破的最低交付底线，若时间紧张，直接砍掉非必要的加分功能，优先保障核心模块的完整可用，杜绝“半拉子工程”。
2.  **每周同步机制**：固定每周日晚开30分钟短会，对齐个人进度、解决技术卡点、微调下周计划，避免问题堆积导致大规模延期。
3.  **卡点快速响应机制**：单个技术问题卡住超过24小时，直接对标开源demo的实现方案，或寻求往届参赛选手/导师的帮助，拒绝死磕浪费时间。
4.  **文档并行机制**：每个模块开发完成的同时，同步完成该模块的文档撰写，绝对不把所有文档堆到最后一周写，避免最终交付期手忙脚乱。
